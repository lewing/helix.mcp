### 2026-03-13: Prefer a narrow Azure.Identity probe with az CLI fallback for AzDO auth
**By:** Dallas
**What:** If we explore Azure.Identity for AzDO auth, keep the existing accessor shape and layer it as `AZDO_TOKEN` env var → explicit Azure.Identity credential(s) for developer auth → current `az account get-access-token` subprocess fallback → anonymous. Do **not** switch to `DefaultAzureCredential`; use a deliberately small credential set and cache Azure.Identity availability/source selection for the session so WSL/libsecret failures do not repeat on every request.
**Why:** The dependency cost is moderate rather than scary: `HelixTool.Core` already carries `Azure.Core` transitively via `Microsoft.DotNet.Helix.Client`, and adding `Azure.Identity` mainly adds the MSAL stack plus a handful of abstractions. The real risk is runtime behavior, not package size — `DefaultAzureCredential` probes too many sources, can add avoidable latency, and muddies the fallback story, while the current az CLI path is already the proven escape hatch for WSL and other broken keychain environments.
